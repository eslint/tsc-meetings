### [Change Request: `includeIgnoreFile` should respect the path to the directory](https://github.com/eslint/rewrite/issues/329)

**TSC Summary:** The proposal is to generalize `includeIgnoreFile` so it resolves patterns relative to the `.gitignore` file location, similar to how git resolves them, rather than relative to the current working directory.

**Discussion Points:**
- `includeIgnoreFile` is currently in `@eslint/compat`, but this new utility could be in a different package since it's no longer related to `eslintrc` only
- Resolving relative to `.gitignore` location was never supported in `eslintrc`
- The team agreed it should live in `@eslint/config-helpers`
- Could potentially use the `basePath` property of config objects
- API design still needs to be figured out, to be discussed on the issue

**Resolution:** The TSC agreed to accept this proposal for adding a utility function to `@eslint/config-helpers`. API design will be discussed on the issue.

### [Infrastructure: Add lockfiles to ESLint repositories](https://github.com/eslint/eslint/issues/20474)

**TSC Summary:** The team revisited the recurring question of whether to add `package-lock.json` to ESLint repositories.

**Discussion Points:**
- The primary benefit of lockfiles is reliably reproducible dev/CI environments
- Without lockfiles, the team gets early detection of ecosystem breakage (e.g., the minimatch v10.2.3 bug would surface immediately in CI)
- With lockfiles, such breakage wouldn't surface until a dependency update PR is merged
- The team would rely on Renovate for dependency updates, which they already largely do
- It's unclear whether Renovate updates transitive dependencies when there are no direct dependency updates
- One option would be to include a lockfile but delete it in CI before install, though this feels messy
- The ESM-only dependencies policy was referenced for context: [ESM dependencies policy](https://github.com/eslint/eslint?tab=readme-ov-file#esm-dependencies)

**Resolution:** The TSC agreed not to add lockfiles, continuing to prioritize early detection of ecosystem breakage.

### [Upgrading eslint-plugin-unicorn in eslint-config-eslint](https://github.com/eslint/eslint/pull/20584)

**TSC Summary:** The question was whether upgrading `eslint-plugin-unicorn` should be allowed given the ESM-only dependencies policy.

**Discussion Points:**
- The team recently introduced a policy about ESM-only dependencies
- Since `eslint-config-eslint` is a package primarily intended for internal use, the policy doesn't need to apply to it

**Resolution:** The TSC agreed that the ESM import rules don't apply to `eslint-config-eslint` because it's intended for own usage only and doesn't pose an ecosystem risk.

### [Add `eqeqeq` to `eslint:recommended`](https://github.com/eslint/eslint/issues/20620)

**TSC Summary:** A request was made to add the `eqeqeq` rule to `eslint:recommended`, which would be a breaking change.

**Discussion Points:**
- `eslint:recommended` mostly includes rules that catch possible errors and disallow legacy syntax
- `eqeqeq` wasn't originally included because it was considered a style preference rather than a best practice
- There's an argument that it's now considered a best practice and maybe should be added
- Reasonable defaults for the rule would need to be discussed
- There's plenty of time before v11 to discuss this

**Resolution:** The TSC agreed to discuss adding `eqeqeq` to `eslint:recommended` in the v11 timeframe. Further discussion will happen on the issue.
