### [refactor: remove is-glob dependency, use minimatch for glob detection](https://github.com/eslint/eslint/pull/20965)

**TSC Summary:** The PR replaces `is-glob` with `minimatch` for glob detection, fixing an inconsistency where patterns like `foo?.js` are treated as an exact match rather than as a glob. The TSC was asked whether this should be treated as semver-patch or semver-minor.

**Discussion Points:**
- @mdjermanovic explained that currently, globs like `foo?.js` or `f?o.js` don't work because `is-glob` returns false, so ESLint treats them as an exact match. With the fix, they would properly match multiple files.
- @fasttime asked whether the fix could cause new files to be enumerated; @mdjermanovic confirmed that it could.
- @nzakas noted the bug has likely been around for a long time; @mdjermanovic added that it probably exists in eslintrc mode in older versions too.
- @nzakas initially leaned semver-patch since people expect the pattern to work this way, but after checking the SemVer policy ("Minor release (might break your lint build): A bug fix in a rule that results in ESLint reporting more linting errors"), he agreed semver-minor was the right fit.
- @mdjermanovic noted an edge case: someone targeting a file that literally has `?` in its path would still enumerate the same files, plus possibly more.
- @mdjermanovic voted for semver-minor; @fasttime didn't feel strongly but was fine with semver-minor.
- @nzakas raised whether the SemVer policy itself should be updated so that any bug fix resulting in more lint errors is semver-minor (not just changes to rules). @mdjermanovic and @fasttime agreed, and suggested using this case (enumerating more files after a bug fix) as an example.

**Resolution:** The TSC agreed to treat the change as semver-minor and to update the SemVer policy to cover any bug fix that might result in more lint errors, not just changes to rules.

**Action Items:**
